# 硬件仓库对接指南 v0.1

> 状态：Android 与硬件/固件联调基线  
> 适用仓库：Android APP 主仓库 `LCSC_android_erp` / VibeRack；硬件仓库 `hardware/PartRack-Hardware`  
> 协议事实源：`docs/智能底盘BLE接口规格_v0.1.md`

本文档说明 Android APP 仓库如何与硬件仓库对接。它不替代 BLE 字节级协议，而是规定两个仓库之间的职责边界、交付物、变更流程和联调验收。

## 1. 对接目标

v1 阶段要跑通一条最小硬件闭环：

```text
扫描或 NFC 识别智能底盘
-> BLE 连接并发现 GATT 服务
-> 读取 Table Info 和绑定表
-> 写入或修改槽位绑定
-> 下发找料/拣料/入库灯光指令
-> 用 table_seq + CRC 校验后更新 APP 本地缓存和库存台账
```

核心原则保持不变：智能底盘的 Binding Table 是智能底盘槽位绑定的事实源；Android 本地 Room 数据库是缓存、富数据和 Stock Ledger，不独立重编号智能底盘槽位。

## 3. 协议基线

Android 和固件当前按 `proto_ver = 0x01` 对接。

必须保持一致的协议项：

| 项 | 当前值 |
|---|---|
| 设备名前缀 | `VBRK-` |
| 开发期 Company ID | `0xFFFF` |
| 槽位数 | 25 |
| 槽位记录 | 16 字节 |
| Light Command | 17 字节 |
| Table Info | 7 字节 |
| Binding Service | `7f4b0001-8d1a-4d45-9a4e-2b4a7c000000` |
| Binding Control Point | `7f4b1001-8d1a-4d45-9a4e-2b4a7c000000` |
| Table Info characteristic | `7f4b1002-8d1a-4d45-9a4e-2b4a7c000000` |
| Light Service | `7f4b0002-8d1a-4d45-9a4e-2b4a7c000000` |
| Light Command | `7f4b2001-8d1a-4d45-9a4e-2b4a7c000000` |
| Light Status | `7f4b2002-8d1a-4d45-9a4e-2b4a7c000000` |

### 3.1 事实源优先级

发生冲突时按以下顺序处理：

1. `docs/智能底盘BLE接口规格_v0.1.md` 是已冻结的 APP/固件共同协议。
2. Kotlin 协议代码必须与该规格 byte-accurate 对齐。
3. 硬件仓库 `docs/app-ble-contract-v0.1.md` 和 `protocol/test-vectors.json` 是硬件侧导出给 APP 的联调材料。
4. 如果硬件仓库文档出现新服务、新字段或新行为，进入 APP 基线前必须同步更新主协议文档和 Kotlin 常量/编解码测试。

当前主协议已纳入 `Device Health Service`（`7f4b0003...` / `7f4b3001...`），payload 为 `battery_pct(1B) + reset_reason(2B LE) + health_flags(1B)`。APP 联调应把 Device Health 与 Binding/Light 一起作为 v0.1 必备自定义服务；BAS/DIS 仍作为可选标准信息处理。

## 4. 共享测试向量

双方必须用同一批测试向量校验协议帧。

当前硬件仓库测试向量：

```text
hardware/PartRack-Hardware/protocol/test-vectors.json
```

关键向量：

| 场景 | Hex |
|---|---|
| `WRITE_ONE` slot 1 / `C1234567` / qty 12 | `10 01 43 31 32 33 34 35 36 37 00 00 0C 00 00 00 18` |
| `READ_ONE` slot 1 | `01 01` |
| `READ_ALL` | `02` |
| `READ_ALL` 结束帧 | `02 00 FF` |
| `FACTORY_RESET` | `F0 A5 A5 5A 5A` |
| `FIND` slot 1 green 30s | `01 01 00 00 00 00 00 00 00 00 FF 00 00 00 00 1E 00` |
| `PICK` slots 1, 7, 25 green 30s | `02 41 00 00 01 00 00 00 00 00 FF 00 00 00 00 1E 00` |
| `OFF` | `00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00` |

Android 侧要求：

- `SmartChassisCodec` 编码结果必须匹配这些 Hex。
- 解析 `READ_ALL` 时必须等待 `02 00 FF` 结束帧。
- `READ_ALL` 完成后必须校验记录数、`slot_count` 和全表 `crc16`，失败不得落账。
- 写类操作遇到 `ERR_FLASH_BUSY` 时退避重试；遇到 `ERR_CRC` 时重新生成记录，不直接更新本地账。

硬件侧要求：

- 所有槽位记录 CRC 使用 CRC-8/MAXIM，覆盖前 15 字节。
- 全表 CRC 使用 CRC-16/CCITT-FALSE。
- 写类操作成功后 `table_seq` 自增并 Notify `Table Info`。
- `INSERT_AT`、`REMOVE_AT`、`MOVE_BLOCK` 的重编号逻辑由 MCU 执行。

## 5. Android 接入流程

### 5.1 连接初始化

建议 Android 连接后按顺序执行：

```text
connectGatt
-> discoverServices
-> read Table Info
-> enable Binding Control Point notify
-> enable Table Info notify
-> enable Light Status notify
-> READ_ALL
-> 校验 table_seq / crc16 / slot_count
-> 更新 Smart Chassis Cache
```

写入 Binding Control Point 前要确保链路已加密。首次 encrypted write 失败时，Android 应触发系统配对或等待 bonding 完成后重试。

### 5.2 业务操作映射

| APP 业务 | BLE 操作 | 本地落账时机 |
|---|---|---|
| Chassis Onboarding | 扫描广告或 NFC URI，读取 Table Info，必要时 READ_ALL | 识别身份并完成校验后创建或更新 Container |
| Slot Binding Workflow | `STOCK_IN` 灯光引导，随后 `WRITE_ONE` 或 `INSERT_AT` | 收到成功响应并校验后更新 Stock Ledger |
| Find-by-Light | `FIND` | 灯控成功不改变库存；若用户确认取料，再用 `SET_QTY` |
| BOM Pick-to-Light | 按 chassis 分组生成 mask，发送 `PICK` | 用户勾选完成后重新下发 mask；数量变更走 `SET_QTY` |
| Slot Reflow | `SORT` 引导，最后 `INSERT_AT` / `REMOVE_AT` / `MOVE_BLOCK` | 结构化操作成功并刷新绑定表后落账 |
| Hardware Restore | NFC 或扫描连接，`READ_ALL` | 完成 CRC 校验后重建最小绑定，再做 Catalog Enrichment |

### 5.3 禁止行为

- Android 不得在智能底盘上独立模拟 `INSERT_AT`、`REMOVE_AT` 或 `MOVE_BLOCK` 后直接落账。
- Android 不得在未收到硬件成功响应时修改智能底盘槽位绑定缓存。
- Android 不得把 `qty = 0` 当作未知数量；未知数量应由 Quantity State 表达。
- Android 不得依赖开发期固定设备名 `VBRK-0000`、固定 `batch_id = 1` 或 Company ID `0xFFFF` 作为量产逻辑。
- 智能底盘槽位不使用打印标签作为身份事实源。

## 6. 联调里程碑

### M1：发现、连接、点灯

验收标准：

- Android 能扫描到 `VBRK-` 设备并解析广播中的 `proto_ver`、`batch_id`、`battery_pct`、`status_flags`、`table_seq`。
- Android 能连接设备、发现 Binding/Light 服务并读取 `Table Info` 和 `Light Status`。
- Android 下发 `FIND` slot 1 后，硬件进入对应 Light Status；接真实灯条时 1 号槽可见点亮。
- 发送 `OFF` 后灯光状态回到 OFF。

### M2：绑定表读写和恢复

验收标准：

- Android 执行 `WRITE_ONE -> READ_ONE`，读回记录与写入记录完全一致。
- Android 执行 `READ_ALL`，收到 25 条记录和 `02 00 FF` 结束帧。
- 写类操作后 `table_seq` 增长，Android 缓存状态更新。
- 设备重启后，写入过的槽位仍可读回。
- Android 可通过 `READ_ALL` 重建智能底盘容器、槽位和最小 Stock Item。

### M3：业务闭环

验收标准：

- 入库绑定：扫码或选择物料，APP 点亮目标槽位，用户确认后写入硬件并落 Stock Ledger。
- 找料：搜索命中智能底盘槽位，APP 点亮实物槽位。
- BOM pick-to-light：按底盘分组下发 `PICK` 掩码，用户完成项目后 APP 重发剩余掩码。
- 断连、配对失败、CRC 错误、flash busy 都有可理解的 APP 错误状态。

## 7. 变更流程

任何会影响 Android/固件互操作的改动都必须走双仓库同步：

1. 在提出改动的仓库先更新协议说明和测试向量。
2. 更新另一仓库的协议常量、编解码、测试或烟测脚本。
3. 提升 `proto_ver` 或在文档里说明为什么不提升。
4. 用 nRF Connect 或 `tools/ble_gatt_smoke_test.py` 完成硬件侧烟测。
5. 用 Android APP 完成 M1/M2 受影响路径复验。

必须同步的改动包括：

- UUID、服务、特征、权限变化。
- 广播字段、NFC URI 字段或 Company ID 变化。
- opcode、status、帧长度、字段偏移变化。
- CRC 算法或 Table Info 格式变化。
- 灯控模式、mask 语义、timeout 规则变化。
- `table_seq` 持久化、自增、Notify 顺序变化。

## 8. 当前待确认项

| 项 | 当前状态 | 对 APP 的影响 |
|---|---|---|
| 真实 WS2812 灯条颜色和槽位映射 | 硬件仓库记录为待真实灯条验证 | APP 可先按 Light Status 联调，最终需实物验收 |
| NFC URI / NT3H2111 | 硬件仓库记录为未完成 APP 可依赖闭环 | APP 保留 URI 路由，M1 可先用 BLE 扫描连接 |
| OTA / Secure DFU | 协议列为目标能力，硬件仓库记录为未完成 | APP 暂不把 OTA 作为 MVP 阻塞项 |
| 真实电池 ADC | 开发板可能返回占位电量 | APP UI 应显示开发态，不做低电量业务判断 |
| Device Health Service | 已纳入主协议，硬件/iOS 可读样例 `64 02 00 00` | Android 常量/编解码应按主协议补齐；BAS/DIS 不得阻塞自定义服务闭环 |

## 9. 联调前检查清单

硬件侧：

- [ ] 设备广播名以 `VBRK-` 开头。
- [ ] Manufacturer Data 可解析，且 `proto_ver = 0x01`。
- [ ] Binding Service 和 Light Service UUID 与主协议一致。
- [ ] `WRITE_ONE -> READ_ONE` 电脑脚本烟测通过。
- [ ] `READ_ALL` 发送 25 条记录和 `02 00 FF` 结束帧。
- [ ] 写类操作后 `table_seq` 自增并发出 Table Info Notify。

Android 侧：

- [ ] Android 12+ BLE 运行时权限完整。
- [ ] 扫描结果能解析硬件广播。
- [ ] 连接后能区分扫描失败、连接失败、服务发现失败、配对失败、协议错误。
- [ ] `SmartChassisCodec` 与共享测试向量一致。
- [ ] `READ_ALL` 超时、结束帧和 CRC 校验处理完整。
- [ ] 写操作成功前不更新智能底盘缓存或 Stock Ledger。
